# 演示流程

本文提供一条可重复的演示路径，方便面试官或 GitHub 访问者从全新部署看到完整 RAG 工作流。

## 演示目标

验证系统可以：

- 摄入一份业务文档。
- 将检索范围限制在当前上传文档和当前会话内。
- 对事实题、数值聚合题、隐含推断题和不可回答题使用不同处理路径。
- 返回答案来源和追踪摘要。
- 执行直接回答 / 标准 RAG / 自适应 RAG 三路对比。

## 启动

```bash
docker compose up --build
```

打开前端：

```text
http://localhost:3001
```

## 示例文档

在 `/documents` 页面上传：

```text
samples/demo_report.md
```

## 示例问题

使用：

```text
samples/demo_questions.md
```

推荐顺序：

1. 询问某个项目的预算和支出。
2. 询问总预算、总支出、总剩余预算等数值聚合。
3. 询问哪个项目最需要调配人手支持。
4. 询问文档中不存在的融资、估值或投资人信息。

## 预期行为

| 问题类型 | 预期路径 | 检查重点 |
| --- | --- | --- |
| 事实查询 | `single_step` | 来源片段中包含答案依据 |
| 数值聚合 | `single_step` + 确定性表格计算 | 数字与表格一致 |
| 隐含推断 | `multi_step` | 答案说明“这是推断”，并给出证据 |
| 不可回答边界 | `multi_step` 或护栏拒答 | 系统拒绝编造缺失事实 |

## 面试讲解重点

- 项目不是简单提示词包装，而是包含摄入、索引、检索、重排、工作流路由、评估和前端交互。
- 自适应路由避免所有问题都走最贵的多步检索路径。
- 会话/文档过滤避免上传文档问答混入历史文档。
- 数值聚合尽量走确定性计算，减少 LLM 自由总结导致的数字幻觉。
- Docker 部署证明项目可以在作者本机之外复现。
