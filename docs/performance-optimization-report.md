# 性能优化报告

日期：2026-07-05

## 摘要

本轮实现了一组可以在无真实 LLM benchmark 的情况下验证的高收益工作流优化：

- 在调用 LLM 分类前增加快速规则分类。
- 在图工作流中增加精确缓存命中短路。
- 增加简单问题的审核快速通道。
- 将 RAGAS 在线评估移出用户请求关键路径。
- 对比评估模式使用 `asyncio.gather` 并发运行多条路径。

完整回归结果：

```text
python -m pytest tests/ -v
68 passed, 1 warning in 5.99s
```

该 warning 是 `.pytest_cache` 写入警告，不是测试失败。

## 优化对比表

| 优化项 | 优化前 | 优化后 | LLM 调用/延迟影响 |
| --- | ---: | ---: | --- |
| 简单查询分类 | 总是调用分类 LLM | 明显简单/复杂的问题由正则规则返回 `simple` / `complex` | 规则命中时减少 1 次分类调用 |
| 缓存命中短路 | 重复问题仍进入完整图路径 | 精确缓存命中后在检索/生成前返回缓存答案 | 命中时跳过检索、生成、审核、RAGAS、护栏 |
| RAGAS 异步化 | 工作流等待 RAGAS 完成 | 工作流先返回，后台继续评估 | RAGAS 不再阻塞用户拿到答案 |
| 审核条件化 | 所有路径都可能进入审核器 | 简单查询默认通过，不调用审核 LLM | 简单查询最多减少 1 次审核调用 |
| 对比评估并行化 | 直接回答、标准 RAG、自适应 RAG 串行 | 三条路径通过 `asyncio.gather` 并发 | 总耗时接近最慢路径，而不是三者相加 |

## 说明

- 缓存集成目前使用请求路径中的精确命中（exact hit）。语义向量缓存查询（semantic embedding lookup）在未命中（miss）时没有立即初始化，因为它会阻塞首个查询测试，并增加目标延迟。
- RAGAS 异步化意味着同一次请求的 HITL 门禁不再同步依赖 RAGAS 分数。质量审核器（reviewer）和安全护栏（guard）仍会参与同次请求决策。
- `src/evaluation/compare.py` 已重写，以移除历史编码损坏字符串，并保持公开的 `run_comparison()` 返回结构不变。
