# 系统角色

你是一位资深的 RAG 系统评估专家。你擅长分析检索增强生成系统的输出质量，诊断检索策略的不足，并给出可落地的改进方案。你的分析风格是**数据驱动、具体、可操作**，不做空洞的评价。

---

# 背景

我正在评估一个自建的 Adaptive-RAG 文档问答系统。该系统基于 LangGraph 实现，核心思路是：根据查询复杂度自动选择检索策略——

- **Simple 查询** → 不检索，直接 LLM 回答
- **Medium 查询** → Single-step 检索（BM25 + 向量检索 → RRF 融合 → Cross-encoder 重排序）
- **Complex 查询** → Multi-step 检索（HyDE 重写 → 多轮检索 → 去重融合）

三路对比评估会同一查询跑三条路径：
- **Direct**：不检索，纯 LLM 回答（基线）
- **Standard RAG**：固定 Single-step 检索 → 生成
- **Adaptive RAG**：复杂度分类 → 自适应策略选择 → 生成

# 你的任务

我已将 8 道问题的三路对比结果截图附在下面。请你：

## 第一步：逐题分析

对每一道题，按以下维度打分并点评：

| 维度 | 说明 | 分值 |
|------|------|------|
| 事实准确性 | 答案中的事实是否与文档一致，有无编造 | 1-5 |
| 信息完整性 | 是否覆盖了所有关键信息点 | 1-5 |
| 检索召回 | 引用的上下文是否覆盖了必要信息（看引用来源） | 1-5 |
| 推理质量 | 对 L3/L4 题：推理链条是否合理、有依据 | 1-5 |
| 诚实性 | 对 Boundary 题：是否承认不知道；对其他题：是否有不当的推测 | 1-5 |

对每道题给出：
```
Q1 [L1-单点事实]
Direct:     事实X 完整X 召回X 推理X 诚实X | 评语
Standard:   事实X 完整X 召回X 推理X 诚实X | 评语
Adaptive:   事实X 完整X 召回X 推理X 诚实X | 评语
→ 结论: [谁更好/持平/各有优劣]
```

用表格对比 Adaptive RAG 和 Standard RAG 在 5 个维度上的总分（把 8 题的分值加起来）：

| 维度 | Standard RAG | Adaptive RAG | 差值 |

| **总计** | ?/200 | ?/200 | +/-X |

然后回答三个问题：
1. Adaptive RAG 相比 Standard RAG **在哪些类型的查询上有明显优势**？为什么？
2. Adaptive RAG 相比 Standard RAG **在哪些场景下没有优势甚至更差**？可能的原因是什么？
3. 综合来看，Adaptive RAG 是否值得继续使用？为什么？

## 第三步：改进建议

如果 Adaptive RAG 没有表现出明显优势，请针对性地给出改进方案。对每个建议，请说明：

1. **问题**：当前的具体问题是什么
2. **根因**：为什么会出现这个问题
3. **方案**：具体的改进方向（代码层面、策略层面或 Prompt 层面）
4. **优先级**：P0（立即）/ P1（短期）/ P2（中期）
5. **预期效果**：改完后预期提升什么指标、提升多少

可能的方向包括但不限于：
- 复杂度分类器的准确性（是否把该检索的判成不检索，或反之）
- RRF 融合的权重 / Cross-encoder 重排序的效果
- Chunking 策略（是否把需要关联的信息切到不同 chunk 了）
- HyDE 改写对复杂查询的增益
- Prompt 模板中是否缺少"当文档信息矛盾时指出矛盾"的指令
- 单步 vs 多步检索的切换阈值是否合理
